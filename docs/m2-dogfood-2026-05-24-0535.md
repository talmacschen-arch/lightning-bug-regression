## 暴露的 backend bug（不阻塞 M2 收尾，单开 followup）

**bug**: `backend/app/runner/orchestrator.py:519` 把 `setup:` 字段当 `list[dict[str, Any]]` 直接迭代，但 YAML schema（design.md §4.1）定义 `setup: list[str]`（每项一句 SQL，复杂步骤才用 dict）。orchestrator 没做 str→dict normalize，第一个 setup step 喂给 `_execute_one_step(step: dict)`，`_step_id` 一行 `step.get("id")` 就 `AttributeError: 'str' object has no attribute 'get'`。

**症状**: API 路径触发的 run 全部 case `status=error, duration_ms=0`（连第一个 setup step 都没进 driver）。

**为什么 M1 dogfood 5/5 PASS 没暴露**: `scripts/run_m1_dogfood.py` (PR #14/#15/#16) 用了 dogfood normalizer 把 setup string 转 `{kind: "sql", sql: <str>}` dict 后再喂 orchestrator。**API 路径 (`POST /runs` → `_load_cases_from_disk`) 绕过这个 normalizer**，直接把 raw YAML dict 喂 orchestrator。M1-10 (API) 跟 M1-11 (dogfood script) 代码路径分叉。

**修法**（next sprint 单独 PR）：
- option A: orchestrator 自己做 str→dict normalize (`{kind: "sql", sql: step}` for string steps，同 dogfood normalizer 逻辑)
- option B: `_load_cases_from_disk` 路径加 normalizer，跟 dogfood script 走同一份
- option C: yaml_loader 在 parse 阶段就 normalize（最一致，但要看 §4.1 schema 是否容忍）

**推荐**: option B，复用 `scripts/run_m1_dogfood.py` 已实现的 normalizer。
