## 7. M5 期间 lesson sticky

### 教训 1 (§14 R24 实战): 本地 ci-gate 必须复 CI 命令 scope
- **触发**: PR #97 M5-3 PR 第一次 ci-gate FAIL — `ruff format --check .` 在 CI 跑全 repo 含 tests/；我本地跑了 `ruff format --check app/` 局部 (只 app 子目录)，没扫到新加的 tests/test_api_cases.py。
- **修法**: 加 ruff format commit + push → ci-gate 重跑 → 绿 → auto-merge 触发。
- **教训**: §14 R24 不只是"跑 ci-gate triplet"，而且**命令完整路径必须复 CI 的命令**。下次提 PR 前在 `backend/` 跑 `ruff format --check .` （从 backend dir）+ `pytest -q`，**不**用 sub-path 偷懒。

### 教训 2 (PR #94 → PR #95 走过的弯路)
- **触发**: foreman 第一次自动 dispatch M5-1 时 specialist 写了带 useEffect+apiFetch / Tailwind 响应式 / shadcn-via-Tailwind-shim 的"全功能"sidebar，9/15 playwright e2e FAIL "element not found"，从 CI 日志无法直接定位（无 artifact 上传）。reviewer agent 也 APPROVE 了（因 reviewer 本机 libgbm 缺，playwright SKIP'd locally）。
- **修法**: 用户决策 kill foreman + 手工重写 minimal Layout（无 useEffect mount / 无响应式 / 无 shadcn / 用 plain CSS），消除嫌疑变量。PR #95 第一次 ci-gate 直接绿。
- **教训**:
  1. **playwright e2e 在 CI 失败但 reviewer 本机 SKIP 是审核 false negative**。reviewer 必须 disclose "我没真跑 e2e" 而不是 APPROVE。
  2. **大改动一次落地多个 sus 变量（mount fetch + responsive + 新 primitive 引入）失败时无法 binary search**。Minimal 优先，验证 OK 后再 incremental 加 feature 是更稳的路径。
  3. **当 CI 失败原因从日志无法定位时**，最快路径不是 deep-debug 而是 reset to minimal 重新 binary search 加回 feature。

### 教训 3 (M5-2 → M5-3 流): "minimal 路径" 自动累积
- M5-2 / M5-3 / M5-4 / M5-5 都沿 M5-1 的 minimal 路径（vitest 单测 + 不写 playwright + plain CSS + no shadcn primitive shims）；5 个连续 PR 全 ci-gate 一次绿（除 #97 ruff fixup）。
- **教训**: pattern 建立后顺势走，比每个 PR 重新决策更可靠。"如果 PR-N-1 的 pattern 让 ci-gate 一把过，PR-N 继续用同 pattern" 是 90% 正确的 default。
