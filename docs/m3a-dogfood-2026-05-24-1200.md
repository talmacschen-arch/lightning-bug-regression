## M3a 暴露的 spec / 工程坑（候选 §14 R / 单独 follow-up）

1. **§5.5.6 canonical 字段顺序与 yaml_loader 不一致**：spec 写 `sessions: []` 但 loader 拒绝空 list。**fix**: §5.5.6 改 `# sessions: {}` 注释提示 "省略时 loader 自动 derive default session"；或 yaml_loader 接受空 list 等价空 mapping。属 design.md / loader spec gap 二选一。
2. **frontend regex vs backend parser 容忍度不一致**（dual code path 轻量版，§14 R26 远亲）：frontend `extractCaseId` 用 `^id:` 严格，但 backend yaml_loader 容忍 leading whitespace。**已修** (#57) `^[\s﻿]*id:`。建议 §14 R26 强化版："frontend 校验 regex 与 backend parser 必须有同一份 reference test，喂相同输入对比 verdict"。
3. **textarea / JSON 链路 silent strip indent 行为不明**：用户粘了 indented yaml，但 yaml 落到磁盘是干净的。可能是 markdown-to-textarea 复制时的 indent block 处理，也可能是 fetch JSON serialize 某层 trim。**暂不修**，但用户经验里这是 footgun（粘什么 ≠ 存什么）。建议在 M3a-7 三段闸门 UI 加一个 "what will be saved" preview block，让用户在 Save 前看到 final yaml。
4. **uvicorn env 配置仍未文档化**（§14 R27 余波）：M3a-3 submit 真打 git/gh 需要 `GH_TOKEN` 与 `LBR_REPO_ROOT` 显式注入 uvicorn process。我手动加了。**TODO**: README 加 "starting dev backend for M3a dogfood" 章节，明列必需 env vars + 起命令模板。
