## Issues found

### F1 — M6-4 skip-list wiring missing (FIXED in this PR)

**Description**: M6-4 PR #115 added `POST /admin/skip-list` but did not
plumb the loaded list through to `orchestrator.run_suite`. Adding a skip
entry had no runtime effect.

**Root cause**: `_execute_run` in `backend/app/api/runs.py` called
`run_suite(...)` without the `skip_list` keyword arg. orchestrator
defaulted `skip_list=[]`.

**Fix**: `_execute_run` now loads `sqlite_store.get_skip_list(sess)`
into the orchestrator-expected `list[dict]` shape and passes via the
`skip_list` kwarg. Failure to load is logged + treated as empty (the
DB read shouldn't ever fail in practice, but we don't want a transient
SQLite hiccup to crash a run).

**Forensic**: this should have been caught earlier — the M6-4 backend
unit test `test_create_skip_list_entry_round_trip` verifies the entry
is persisted but doesn't run a run and assert the case was skipped.
Added followup: a dedicated integration test for the wiring would have
caught it. (Out of scope for M6-6 dogfood; logged as candidate §14 R
followup.)

### F2 — Step 02 (precondition) artifact missing from list

**Description**: Artifact list for run #15 has files for step_idx 0/1
(setup) and 3+ (ZQL queries), but no step_idx 2 entries.

**Hypothesis**: step 2 is the ES health-check shell step
(`su - gpadmin -c 'curl ...'`). The `su` subshell inherits stdout/stderr
but the orchestrator's `_write_artifact` writes whatever the
`execute_shell_step` driver returned. Possibly the redirect-to-su
swallowed stdout in a way that made `step_result.stdout` empty (then
`_write_artifact` returns None for empty content).

**Impact**: low — the case passed (so the assertion `stdout_contains:
'"status" : "green"'` evaluated against something non-empty). Most
likely the result was captured but not persisted as an artifact file.

**Followup**: not in M6-6 scope. Investigate next time we touch the
shell driver (or M7 onward).
